# workflow — 進め方

関連: [ADR-0005](../decisions/0005-one-session-one-repo.md)（1セッション=1担当リポ）

## 1セッション = 1担当リポ

- 書き込みは担当リポ内だけ。他リポは**読むのは可**、書くなら Issue を立てて引き継ぐ。
- これは散文の約束ではなく scope-guard hook で物理的に強制される（[guards](guards.md)）。

## 1セッション = 1Issue（基本）

定型フロー: **ピック → 計画提示 → （承認）→ 実装 → コミット → 引き継ぎ/終了処理**。
カスタムラベルを増やさず、Issue本文の「Blocker」節で着手可否を表す。

## 踏み込みすぎを止める（最初の1メッセージ）

- セッション冒頭で「これから何をするか」を明示。**調査だけなら「実装はまだしない」と宣言**する。
- 設計余地の大きい変更は、まず読み取り専用で計画を立て、**人が承認してから実装**へ進む。
- 「進めて」「やって」は作業継続の許可であって、コミット/破壊的操作の承認ではない。確証がなければ確認する。

## 詰まったときの戻し方（破壊は最終手段）

1. `git diff` で差分確認 → 2. ファイル単位で戻す（`git checkout -- <file>`）→ 3. 退避（`git stash`）→ 4. 全捨て（`git reset --hard`）は最終手段で、**自動実行しない**。

## 終了プロトコル（変化 → 反映先）

| 変化 | 反映先 |
|---|---|
| 横断・恒久の決定 | docs-hub の `decisions/` に起票（作業リポからは Issue で依頼） |
| 設計のずれ | docs-hub の該当 `design/*.md`（同上） |
| タスク完了 | 該当 Issue をクローズ |
| 引き継ぎ | Issue コメント |
| リポ固有の設計判断 | そのリポの `docs/adr/` |
