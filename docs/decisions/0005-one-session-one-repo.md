# ADR-0005: 1セッション = 1担当リポ制を採用する

- ステータス: Accepted
- 日付: 2026-06-22
- 関連: [ADR-0001](0001-multi-repo.md), [workflow](../dev-environment/workflow.md), [guards](../dev-environment/guards.md)

## 決定

1つの作業セッション（エージェント or 人間）は、**1つのリポの担当者**として振る舞う。書き込みは担当リポ内だけに限定し（読み込みは他リポ可）、他リポの変更が必要になったら**その場で書かず Issue を立てて引き継ぐ**。

## 背景

マルチリポ（[ADR-0001](0001-multi-repo.md)）では、ある作業のついでに別リポを書き換えると、(1) その別リポの規約・ガード・文脈が読まれないまま編集される、(2) 担当境界が崩れてレビュー単位が壊れる、(3) 横断SoT（docs-hub）が副作用で書き換わる、という事故が起きる。

## 帰結

- **良い**: 作業ディレクトリ・CLAUDE.md・ガード設定が常に一貫する。コミット/Issue が自然にリポ単位で揃う。
- **強制**: この境界は散文だけでは守られない。担当外パスへの書き込みは **scope-guard hook（block型）** で物理的に拒否する（[guards](../dev-environment/guards.md)）。所有関係のある隣接リポだけ `cross-repo-allowlist` で例外許可する。
- **横断SoTの保護**: 作業リポ側は「Issueを上げるか/上げないか」だけを判断し、決定として記録するか等の体裁判断は docs-hub 側が行う（末端で記録形式を勝手に作らせない）。

## 検討した代替案

- **境界なし（どこでも編集可）**: 個人開発では一見楽だが、ロジック/UIの責任分界が崩れ、`core` の純粋性（[ADR-0002](0002-pure-core-engine.md)）が守られなくなる。却下。
