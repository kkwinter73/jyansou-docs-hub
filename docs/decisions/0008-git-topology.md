# ADR-0008: git topology — 各サブリポを独立 git リポにし、親はただのディレクトリ

- ステータス: Accepted
- 日付: 2026-06-22（2026-06-22 改訂: 親を git リポにしない方針に変更）
- 関連: [ADR-0001](0001-multi-repo.md), [ADR-0005](0005-one-session-one-repo.md)

## 決定

`docs-hub` / `core` / `web` を**それぞれ独立した GitHub リポジトリ**にする（リモートも別々）。親ディレクトリ `jyansou/` は3リポを横並びに clone する**ただの入れ物**であり、**git リポジトリにはしない**。ワークスペースの地図・クローン手順は SoT である [docs-hub の README](../../README.md) が持つ。

## 背景

[ADR-0001](0001-multi-repo.md) でマルチリポを選んだ。git topology には選択肢がある: (A) 各リポ独立、(B) git submodule、(C) モノレポ + workspaces。当初は「親も薄い git リポにしてワークスペースREADMEを持つ」案を採ったが、(1) 親リポの中身は地図だけで、その地図は本来 docs-hub が持つべき重複、(2) GitHub リモートが1つ余分に増え管理対象が増える、ため**親を非リポ化**した。

## 帰結

- **良い**: 各リポが独立した履歴・ブランチ・Issue・ガード・リモートを持つ。[ADR-0005](0005-one-session-one-repo.md)（1セッション1リポ）と素直に対応。親に余計なリポ/リモートが無く、地図は docs-hub の1か所に集約される。
- **悪い / コスト**: ネストした git リポではないが、親ディレクトリ自体はバージョン管理されない（並べ方の手順は docs-hub に明記して担保）。リポ間のバージョン整合は手動。
- **運用**: `web` → `core` の依存は、当面 npm の **ローカルファイル依存/相対 alias**（`../core`）で繋ぐ。公開レジストリには出さない。

## 検討した代替案

- **親も薄い git リポにする（旧案）**: 地図が docs-hub と重複し、リモートが余分に増えるため却下（本ADRの改訂で破棄）。
- **git submodule (B)**: バージョン固定はできるが、個人開発では submodule 更新の手間と事故が多い。却下。
- **モノレポ + pnpm workspaces (C)**: 配線は最も楽。ただし `docs-hub` という非コードSoTの独立性と、リポ単位の独立性（[ADR-0005](0005-one-session-one-repo.md)）を最優先し、今回は不採用。`server` 追加等で関心が増えたら再検討の余地を残す（その場合ADRで上書き）。
