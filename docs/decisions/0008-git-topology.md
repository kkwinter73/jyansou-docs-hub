# ADR-0008: git topology — 各サブリポを独立 git リポにし、親はただのディレクトリ

- ステータス: Accepted
- 日付: 2026-06-22（2026-06-23 改訂: 親を git リポにしない方針に変更／粒度拡充）
- 関連: [ADR-0001](0001-multi-repo.md) / [ADR-0005](0005-one-session-one-repo.md)

## コンテキストと問題提起

マルチリポ（[ADR-0001](0001-multi-repo.md)）を選んだ。その上で git の物理構成を決める必要がある。選択肢は (A) 各リポを独立 git リポにする、(B) git submodule で結合する、(C) モノレポ + workspaces。あわせて「親ディレクトリ `jyansou/` を git リポにするか」も決める。

当初は「親も薄い git リポにしてワークスペースREADMEを持つ」案を採ったが、(1) 親リポの中身は地図だけで、その地図は本来 docs-hub が持つべき重複、(2) GitHub リモートが1つ余分に増え管理対象が増える、ことが判明したため本ADRで方針を改訂した。

## 決定要因

- 各リポが独立した履歴・ブランチ・Issue・ガード・リモートを持てること（[ADR-0005](0005-one-session-one-repo.md) と一致）
- 真実源（地図・クローン手順）が1か所（docs-hub）に集約され重複しないこと
- 管理対象（リモート）を最小に保てること
- リポ間の依存配線が現実的な手間で済むこと

## 決定結果

### 各サブリポを独立した GitHub リポジトリにする

- **根拠**: 独立した履歴・ブランチ・Issue・ガードを持て、[ADR-0005](0005-one-session-one-repo.md)（1セッション=1リポ）と素直に対応する。
- **影響**: `jyansou-docs-hub` / `jyansou-core` / `jyansou-web` を別リモートで持つ。

### 親ディレクトリは git リポジトリにしない

- **根拠**: 親の中身（地図）は docs-hub と重複し、余分なリモートが増えるだけ。
- **影響**: 親 `jyansou/` は3リポを横並びに clone する**ただの入れ物**。並べ方・クローン手順は docs-hub の README が持つ。

### リポ間依存はローカル相対参照

- **根拠**: 公開レジストリに出す段階ではないため。
- **影響**: `web` → `core` は npm のローカルファイル依存／相対 alias（`../core`）で繋ぐ（[ADR-0003](0003-frontend-stack.md)）。

## 利点

- 各リポが独立して進化でき、セッション境界（[ADR-0005](0005-one-session-one-repo.md)）と物理構成が一致する。
- 親に余計なリポ/リモートが無く、地図は docs-hub の1か所に集約される。
- 将来 `server` を足すときも、同じ標準形のリポを1つ増やすだけ。

## トレードオフ

- 親ディレクトリ自体はバージョン管理されない。→ 並べ方の手順を docs-hub に明記して担保する。
- リポ間のバージョン整合は手動。→ 当面は相対参照で十分。公開・配布段階になれば再検討。

## 不採用の選択肢と根拠

| 選択肢 | 不採用理由 |
|---|---|
| **親も薄い git リポにする（旧案）** | 地図が docs-hub と重複し、リモートが余分に増える。本ADRの改訂で破棄。 |
| **git submodule (B)** | バージョン固定はできるが、個人開発では submodule 更新の手間・事故が多い。 |
| **モノレポ + pnpm workspaces (C)** | 配線は最も楽だが、docs-hub（非コードSoT）の独立性とリポ単位の責任分界を最優先し不採用。`server` 追加等で関心が増えたら再検討の余地を残す（その場合ADRで上書き）。 |
