# ADR-0001: マルチリポ構成を採用する（docs-hub / core / web）

- ステータス: Accepted
- 日付: 2026-06-22（2026-06-23 粒度拡充）
- 関連: [ADR-0002](0002-pure-core-engine.md) / [ADR-0005](0005-one-session-one-repo.md) / [ADR-0008](0008-git-topology.md) / [architecture](../design/architecture.md)

## コンテキストと問題提起

日本リーチ麻雀のブラウザゲームを新規構築する（[ADR-0004](0004-ruleset-riichi-full.md)）。シングルプレイ（対CPU）を先に作り、将来のオンライン対戦（権威サーバ）も見据える。

このプロジェクトには性質の異なる3種類の関心が同居する。(1) **ルールロジック**（牌・和了判定・役・点数・局進行）は複雑でテストが必須、かつ将来サーバとクライアントで共有したい。(2) **UI**（盤面描画・入力・アニメ）はブラウザ環境に強く依存する。(3) **設計知識**（決定記録・API契約・進め方）は複数リポ／複数人にまたがる。

これらを単一の置き場所に混ぜると、UIの都合でロジックが汚れ、横断知識が各所に散って真実源が曖昧になり、将来ロジックを切り出せなくなる。コードと知識の配置方針を最初に決める必要がある。

## 決定要因

- ルールロジックの**純粋性**（フレームワーク非依存）を構造的に強制できること（[ADR-0002](0002-pure-core-engine.md)）
- 横断知識の**単一の真実源（SoT）** を1か所に確立し、副作用で書き換わらないこと
- 担当境界が明確で、レビュー単位・作業履歴がリポ単位に自然に揃うこと（[ADR-0005](0005-one-session-one-repo.md)）
- 将来のオンライン化で `core` をサーバへ無改造で移植できること
- 1人〜小規模で長期運用でき、配線の手間が過大にならないこと

## 決定結果

知識とコードを**役割の異なる独立リポ**に分離し、同一の親ディレクトリに横並びにして相対パス `../` で参照する（git topology は [ADR-0008](0008-git-topology.md)）。

### docs-hub（横断SoT・コードを持たない）

- **根拠**: 複数リポ／他人が参照する恒久知識（決定記録・設計契約・共通ハーネス部品の原本）を1か所に集約する。コードを持たないことで「設計の真実源」と「実装」を物理的に分離する。
- **影響**: ADR・設計書・`templates/.claude/`（共通hooks/rules/skills原本）・索引生成スクリプトを配置。各作業リポからは読み取り専用で参照し、書き込みは行わない。

### core（麻雀エンジン・純粋ロジック）

- **根拠**: ルール判定をUI・乱数・時刻・ネットワークから切り離し、Node（サーバ）でもブラウザでも同一に動く純粋関数として実装する（[ADR-0002](0002-pure-core-engine.md)）。
- **影響**: TypeScript。実行時依存を持たない（`package.json` の dependencies は空）。公開APIは `src/index.ts` に集約し、内部表現を外に漏らさない（[architecture](../design/architecture.md)）。

### web（ブラウザフロント）

- **根拠**: 描画・入力・CPU起動・seed生成といった環境依存の責務を担い、ルール判定は必ず `core` に委譲する。
- **影響**: React + Vite（[ADR-0003](0003-frontend-stack.md)）。`core` を相対依存で取り込む。

将来追加候補（未作成）: `server`（オンライン対戦の権威サーバ。`core` を共有）、`e2e`（E2Eテスト）。

## 利点

- `core` の純粋性を「依存に置けない」構造で守れる（散文の約束に頼らない）。
- 横断知識が docs-hub の1か所に集中し、各作業リポは自己完結した同じ標準形になる。
- 担当境界が物理的に明確で、`A` の担当者が `B` を勝手に書き換える事故を防げる（[ADR-0005](0005-one-session-one-repo.md)）。
- 将来の `server` 追加時、`core` をそのまま共有でき二重実装を避けられる。

## トレードオフ

- リポ間のバージョン整合・依存配線が手動になる。→ 当面 `web` は `../core` をローカル相対依存で参照し、公開レジストリには出さない（[ADR-0008](0008-git-topology.md)）。
- 個人開発では「1つのPRで全リポ横断」ができず、Issue経由の引き継ぎが必要になる場面がある。→ これは責任分界を守るための意図的なコスト。

## 不採用の選択肢と根拠

| 選択肢 | 不採用理由 |
|---|---|
| **単一リポ（モノレポ）** | 配線は楽だが、`core` の純粋性を散文の約束でしか守れず、UI依存の混入を構造的に防げない。 |
| **モノレポ + pnpm/turbo workspaces** | 中間案で配線は最も楽。ただし `docs-hub`（非コードSoT）の独立性とリポ単位の責任分界を最優先し不採用。`server` 追加で関心が増えたら再検討の余地を残す。 |
| **git submodule で結合** | バージョン固定はできるが個人開発では更新の手間・事故が多い（[ADR-0008](0008-git-topology.md) 参照）。 |
