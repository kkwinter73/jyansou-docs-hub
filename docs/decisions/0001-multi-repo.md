# ADR-0001: マルチリポ構成を採用する（docs-hub / core / web）

- ステータス: Accepted
- 日付: 2026-06-22
- 関連: [ADR-0002](0002-pure-core-engine.md), [ADR-0008](0008-git-topology.md), [architecture](../design/architecture.md)

## 決定

知識とコードを **役割の異なる独立リポ**に分離する。横断SoTを持つ `docs-hub`、麻雀エンジンの `core`、ブラウザフロントの `web` を同一の親ディレクトリに横並びにし、相対パス `../` で参照する。

## 背景

麻雀ゲームは「ルールロジック（複雑・テスト必須・将来サーバ共有）」「UI（描画・入力・アニメ）」「設計知識（決定・契約・進め方）」という性質の違う関心が混在する。これらを単一の置き場所に混ぜると、(1) UIの都合でロジックが汚れる、(2) 横断知識が各所に散って真実源が曖昧になる、(3) 将来オンライン対戦でサーバとロジックを共有するときに切り出せない、という問題が起きる。

## 帰結

- **良い**: 関心が物理的に分離され、`core` はフレームワーク非依存を強制できる（[ADR-0002](0002-pure-core-engine.md)）。横断知識は `docs-hub` の1か所に集約される。担当境界が明確になる（[ADR-0005](0005-one-session-one-repo.md)）。
- **悪い / コスト**: 個人開発ではリポ間のバージョン整合・依存配線の手間が増える。→ 当面は `web` から `core` を**相対パス/ローカル依存**で参照し、パッケージ公開はしない。git topology は [ADR-0008](0008-git-topology.md) で別途決める。
- **強制**: 担当外リポへの書き込みは scope-guard hook で物理的に止める（[guards](../dev-environment/guards.md)）。

## 検討した代替案

- **単一リポ（モノレポ）**: 配線は楽だが、`core` の純粋性を散文の約束でしか守れず、UI依存が混入するリスクが高い。却下。
- **モノレポ + workspaces（pnpm/turbo）**: 中間案。将来 `server` 追加で関心が増えたら再検討の余地あり。現時点では `docs-hub` という非コードSoTの分離を最優先し、素朴なマルチリポを採る。
