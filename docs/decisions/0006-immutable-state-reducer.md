# ADR-0006: ゲーム状態は不変データ + 遷移関数（reducer）で表現する

- ステータス: Accepted
- 日付: 2026-06-22
- 関連: [ADR-0002](0002-pure-core-engine.md), [ADR-0007](0007-seedable-prng.md), [game-flow-design](../design/game-flow-design.md)

## 決定

`core` のゲーム状態 `GameState` は **不変（immutable）** とし、状態遷移は純粋関数 `apply(state, action) -> { state, events }` で表現する。状態を破壊的に変更しない。プレイヤーの操作・自動進行はすべて `Action`（明示的なデータ）として表現する。

## 背景

麻雀は「ツモ・打牌・チー/ポン/カン・リーチ宣言・和了・流局」といった離散イベントで進む典型的な状態機械。可変状態を各所で書き換える設計は、(1) どこで状態が壊れたか追えない、(2) UIの楽観更新とサーバ確定のズレを扱えない、(3) リプレイ/牌譜を再現できない、という問題を生む。

## 帰結

- **良い**:
  - **決定論**: `(初期state, actionの列)` から最終stateが一意に再現できる → 牌譜リプレイ・サーバ権威化（[ADR-0002](0002-pure-core-engine.md)）・テストが容易。
  - **検証**: 各 `apply` は副作用なしで単体テストできる。不正なActionは `apply` が拒否（理由付きで返す）。
  - **UI**: `web` は確定stateを描画するだけ。楽観更新は「予測apply→サーバ確定で置換」で扱える。
- **悪い / コスト**: 毎遷移で新オブジェクトを作るため素朴には割当が増える。→ 麻雀は1局あたりの遷移数が小さく実用上問題ない。必要なら構造共有（Immer等）を後で局所導入（その時ADR追加）。
- 乱数を伴う遷移（牌山生成・ツモ）は、PRNGの状態も `GameState` に内包して持ち回る（[ADR-0007](0007-seedable-prng.md)）。これにより `apply` の純粋性を保つ。

## 検討した代替案

- **OOPの可変 `Game` オブジェクト（メソッドが内部状態を変更）**: 直感的だが上記の決定論・リプレイ・サーバ共有の利点を失う。却下。
- **イベントソーシング（イベントだけ保存しstateは都度畳み込み）**: reducer設計の自然な拡張。`apply` が `events` を返す形にしておき、必要になれば牌譜＝Action列の永続化で実現できる。今は state を直接持つ。
